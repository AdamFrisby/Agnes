# Session forking & replay

| | |
|---|---|
| **Category** | Sessions |
| **Plugin surface** | Core host feature — the event-sourced session log already provides most of what forking needs; optional `IForkCapableAdapter` capability for agents whose underlying CLI can fork a conversation natively |
| **Priority** | P1 — foundational groundwork for `../connectivity/03-session-handoff.md` |
| **Rough effort** | L |

## Background

Long coding-agent sessions accumulate a lot of context: files read, decisions made, dead ends explored. Two situations come up constantly once a session runs for a while:

1. You want to try a different approach from an earlier point without losing the work the session has already done — "what if I'd asked it to use library X instead of Y back at that message" — without destroying the session that already made progress down the original path.
2. A session has grown long enough that continuing to pay context-window cost on its full history is wasteful, and a fresh session seeded with a condensed recap of "what we've established so far" would be cheaper and faster to keep working in.

Both are the same underlying operation: create a new session whose starting context is derived from an existing session's history up to some point, after which the two sessions evolve completely independently. That's forking.

Agnes is well positioned to build this cleanly because of a decision already made in the host: every agent update is normalized into a `SessionEvent` and appended to that session's own per-session SQLite log with a monotonically increasing sequence number (see the "event-sourced session model" section of `/work/docs/architecture.md`). That gives forking a precise, cheap-to-reason-about definition: **a fork is a new session whose initial context is derived from the parent's event log up to sequence N**, after which the two logs are independent. Forking doesn't need a new kind of write to the storage model — it's a read of existing history plus a new session, both of which Agnes already knows how to do.

## Current state in Agnes

Agnes has no forking today. `IAgentSession` (`/work/src/Agnes.Abstractions/Agent.cs`) exposes `PromptAsync`, `CancelAsync`, `RespondToPermissionAsync`, and mode switching, but no operation that starts a new session seeded from another session's history. `AgentSessionOptions.ResumeSessionId` reconnects the *same* logical session under a new `IAgentSession` instance after something like a host restart — a different problem: it restores continuity for one conversation, it doesn't branch a conversation into two.

## Proposed design

```csharp
// Agnes.Abstractions
public interface IForkCapableAdapter
{
    /// <summary>Whether the underlying CLI can fork natively rather than needing replay.</summary>
    ForkSupport Support { get; }
    Task<IAgentSession> ForkNativeAsync(IAgentSession parent, string? atMessageEventId, CancellationToken ct = default);
}

public enum ForkSupport { None, ForkAtLatest, ForkAtMessage }
```

`SessionManager` gains `ForkSessionAsync(sessionId, atSequence, ForkSeedStrategy strategy)`:

1. **Prefer provider-native forking when it exists.** If the parent's `IAgentAdapter` implements `IForkCapableAdapter` and reports `ForkAtMessage` (or `ForkAtLatest` when `atSequence` is the current head), delegate to `ForkNativeAsync`. A CLI that keeps its own internal conversation state can branch it more cheaply and more faithfully than Agnes reconstructing an equivalent conversation from the outside — no reason to reinvent that when it's available.
2. **Fall back to replay-fork when it isn't**, and this path must always work regardless of which CLI is behind the session, since Agnes's whole premise is running many different agent CLIs uniformly and none of them should be second-class for a feature this central. Replay-fork reads the parent's event log up to `atSequence` and builds a seed per `ForkSeedStrategy`:
   - `RecentMessages` (v1): the transcript tail, verbatim. Cheap, provider-agnostic, needs nothing the log doesn't already store — this alone is enough to ship the feature.
   - `SummaryAndRecent` (later addition): a structured summary of the full parent history up to the fork point, generated by a one-shot prompt to the same agent, with the recent-message tail appended after it. This only pays for itself once sessions get long enough that a verbatim replay would burn a large share of the child's context budget just restating history — worth building once there's a concrete case demanding it, not speculatively up front.

   The seed is sent as the child session's first `PromptAsync` call, and it is deliberately **not** recorded as a visible `SessionEvent` in the child's own log. The seed is a mechanism for transferring context, not something a person typed; showing it in the transcript as if the user pasted in the whole prior conversation would be confusing and would clutter every forked session's history with an implementation artifact. The child also gets its own fresh provider-side session — its own tool state, its own permission grants — because forking branches *conversation context*, not the live process or its side effects; reusing the parent's live process would risk leaking one branch's file edits or granted permissions into the other.
3. **The new session's event log begins with a cross-reference marker** — a new `SessionEvent` kind, `ForkedFromEvent(ParentSessionId, ParentSequence)` — so the transcript UI can render the parent's history read-only above a "Forked from…" divider by resolving the marker at render time, rather than copying the parent's event rows into the child's own log. This keeps the design consistent with the event-sourced model's core property (one log is the single source of truth for its own history): copying rows would create a second copy of the parent's history that could silently drift if the parent is later edited, exported, or deleted. The cost is that rendering a forked child now sometimes needs to read from *two* session logs instead of one — a genuinely new read path, called out explicitly below rather than assumed to fall out for free.
4. **Where you fork from determines whether the origin text becomes an editable draft.** Forking at a user message forks immediately *before* that message and restores its text as an editable draft in the new session's composer — this is the common case ("I want to try sending this differently"), and pre-filling it removes a copy/paste step. Forking at an assistant or tool-output message forks immediately *after* it, with no draft, since there's no message text at that point to re-edit.

## Acceptance criteria

- Given a session with at least one prior turn, forking at any specific message produces a new, independent session whose visible history shows the parent's transcript up to that point, read-only, above a clear "forked from" boundary.
- Given an agent whose adapter does not implement `IForkCapableAdapter`, forking still succeeds via replay-fork — no agent kind is ever unable to fork.
- Given a fork at a user message, the forked message's text is pre-filled as an editable draft in the new session's composer; given a fork at an assistant or tool message, the composer starts empty.
- The seed content used to prime a replay-forked session never appears as an ordinary chat bubble in the child's own transcript.
- After forking, sending further messages in the parent session has no effect on the child's transcript, and vice versa — the two event logs are fully independent from the fork point onward.
- Forking a session that is itself a fork succeeds and correctly renders the read-only history chain back through both ancestors.
- Deleting or archiving a parent session after a child has been forked from it does not break the child's transcript view — the marker-resolution path degrades gracefully (e.g. shows "original session no longer available") rather than erroring.

## Open questions

- Does any of Agnes's current or planned adapters expose a native fork verb at all today? This needs checking against each CLI's actual protocol surface before assuming `IForkCapableAdapter` has any real implementations at launch — replay-fork alone may need to be the entire v1.
- Cross-session reads for `ForkedFromEvent` resolution are a new capability for the transcript-rendering path (today it only ever reads one session's log at a time). Worth scoping as its own implementation task rather than assuming it falls out of the new event kind for free.
