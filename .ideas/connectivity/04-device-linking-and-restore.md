## Background

There are at least three distinct situations a user can be in when they need to authenticate a new device or process against Agnes, and they call for genuinely different defaults and UI:

1. **Adding a new device to an account you're already signed into elsewhere** — e.g. you're already paired on your desktop and want to add your phone. The device on hand (the phone) needs a way to prove "I should be trusted the same way my desktop is," typically by scanning something the already-trusted device displays.
2. **Recovering access after losing every device** — e.g. your only paired laptop was lost, stolen, or wiped, and you're setting up a brand-new device with nothing paired yet. This is not "add a device," because there is no already-trusted device to vouch for the new one; it requires a different credential entirely (something you set aside in advance, or an identity check independent of any previously-paired device).
3. **Authorizing a headless process, not a human-facing device** — a daemon or terminal session that needs to act on the account but isn't something a person is going to "sign in" to in the usual sense.

Conflating these into one generic "here's a token" flow is a real usability trap: the UI for "scan this code with your phone" makes no sense for a headless terminal, and neither makes sense for "I lost my phone and have nothing to scan with." Each needs a UI entry point whose default matches what the user is actually trying to do.

There's also a sharper reason this matters for Agnes specifically going forward: once `../security/01-end-to-end-encryption.md` lands, each device will hold trust material (a pinned certificate/keypair) that isn't recoverable from the host alone by design — that's the whole point of that model. At that point, losing your only device stops being a minor inconvenience and becomes a genuine permanent-data-loss risk unless there's a deliberate recovery path. "Restore account" changes from a nicety to a hard requirement the moment that lands.

## Current state in Agnes

Per `/work/docs/deployment.md`, Agnes already has three real ways to authenticate a device against a host:

- **Pairing code** — a short code shown in the host's logs/UI, entered on a new client, exchanged via `POST /pair` for a durable per-device bearer token.
- **GitHub sign-in (device flow)** — strong identity-based auth via a GitHub OAuth App, checked against an allowlist of users/orgs/teams, with no shared secret and no callback URL needed.
- **Keypair sign-in (offline)** — SSH-`authorized_keys`-style: a client-held P-256 keypair, with the public key added to the host's authorized-keys file, authenticated by signing a single-use challenge.

`GET /auth/methods` already exists and lets a client discover which of these a given host supports. `GET /devices` and `DELETE /devices/{id}` already exist for listing and revoking paired devices.

What's missing is the **product-level distinction** between the three situations described above. Today all three mechanisms are exposed as one flat list of "ways to get a token," with no signal to the client about which real-world situation each one is meant for — a user with no devices at all and a user adding a second device see the same flat list of options. There's also a concrete, narrow bug class worth fixing at the same time: a QR code or deep link generated by a host that only knows its own LAN-local or loopback address (`http://127.0.0.1:...` or a private LAN IP) is useless when scanned by a phone on a different network — the host needs to know and advertise an address that's actually reachable by the device scanning the code.

## Proposed design

This is mostly a client-UX and host-endpoint-semantics change, not a new heavyweight interface — but it's the natural place to pluginize the three existing auth mechanisms behind the `IAuthMethodProvider` interface named in `../00-plugin-architecture.md`, so new methods (e.g. an enterprise SSO provider, see `../security/02-enterprise-auth.md`) can be added later without touching client UI logic:

```csharp
public interface IAuthMethodProvider
{
    string MethodId { get; }              // "pairing-code" | "github-device-flow" | "keypair"
    string DisplayName { get; }
    AuthFlowKind Kind { get; }            // NewDevice | RestoreAccount | ConnectTerminal
    Task<AuthResult> AuthenticateAsync(AuthContext context, CancellationToken ct = default);
}
```

Each of today's three mechanisms becomes an `IAuthMethodProvider` implementation, tagged with which of the three real-world situations (`AuthFlowKind`) it's meant to serve. `Agnes.Host`'s existing `GET /auth/methods` endpoint reports each provider's `AuthFlowKind` alongside its id, so the client groups them into the right UI bucket (e.g. "Add this device" vs. "I lost access, help me recover" vs. "Authorize a headless process") instead of presenting one undifferentiated list regardless of what the user is actually trying to do.

For the loopback/LAN-address bug: the host needs to advertise an address that's actually reachable by the device that will scan the code, not just whatever address it's bound to locally. This is the same problem `01-relay-and-tunneling.md`'s `TransportEndpoint` already solves for ordinary connections — reuse it here rather than inventing a second "what's my real address" mechanism: the pairing/QR flow should ask the active `ITransportProvider` for the host's externally-reachable `TransportEndpoint` and encode *that* into the QR/deep link, falling back to an explicit operator-configured override (fitting into Agnes's existing `Agnes:` configuration namespace, e.g. `Agnes:PublicUrl`) for cases the transport can't infer automatically, such as a reverse proxy in front of the host.

## Acceptance criteria

- **AC1** — `GET /auth/methods` reports an `AuthFlowKind` for every configured auth method, and the client UI groups available methods into distinct "add a device," "restore account," and "authorize a headless process" sections rather than one flat list.
- **AC2** — A QR code or deep link generated by a host reachable only through a relay or a reverse proxy encodes an address that a client on a different network can actually resolve and connect to — verified by generating a pairing QR on a host with no public LAN-reachable address and confirming a client on a separate network can complete pairing by scanning it.
- **AC3** — A host with a configured `Agnes:PublicUrl` override always uses that value for QR/deep-link generation, regardless of what the transport layer would otherwise infer.
- **AC4** — Given a user with zero paired devices, when they open the client for the first time, then they are shown restore-account and new-account-setup options, not a flow that assumes an already-trusted device exists to scan a code.
- **AC5** — Revoking a device via the existing `DELETE /devices/{id}` continues to work unmodified after auth methods are pluginized behind `IAuthMethodProvider`.
- **AC6** — All three of today's existing auth mechanisms (pairing code, GitHub device flow, keypair) continue to authenticate successfully after being wrapped as `IAuthMethodProvider` implementations — no regression in existing pairing behavior.

## Open questions

- A "restore with a recovery credential" flow only becomes meaningful once there's a client-side secret that isn't recoverable from the host alone (see `../security/01-end-to-end-encryption.md`) — sequence this after that work lands, since before then there's no client-held secret that actually needs restoring.
- Should a device-management screen (backed by the existing `GET /devices` / `DELETE /devices/{id}` endpoints) be surfaced in the client as part of this work, or is that better scoped as its own small follow-up? The endpoints already exist server-side; this is purely a client-UI sequencing question.
